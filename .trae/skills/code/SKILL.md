# code 技能

## 技能目标

围绕用户当前学习主题或指定内容，提供用于理解机制、流程或业务逻辑的代码示例（Code Sample）。代码的重点是解释思路与结构，而不强制要求可以直接运行。

## 适用场景

当出现以下情况时，应调用本技能：

- 用户希望把抽象概念转化为更直观的代码表达。
- 用户已经理解了部分理论，想看对应的伪代码或示例代码。
- 用户想通过代码理解算法流程、模块关系或业务逻辑。
- 需要演示某个机制如何从概念层落到实现层。

## 输出目标

本技能生成的代码示例应满足以下目标：

- 帮助用户理解语法结构。
- 帮助用户理解每一步逻辑在做什么。
- 帮助用户理解业务目标与代码结构之间的映射关系。
- 帮助用户从“会看公式”过渡到“会读实现思路”。

## 重要约束

1. 代码不要求可直接运行。
2. 代码必须足够清晰，能够描述核心业务逻辑或算法流程。
3. 注释必须充分，不能只给极少量说明。
4. 注释应同时覆盖语法层、逻辑层和业务层。
5. 如果存在更易理解的简化写法，应优先选择清晰而非炫技。
6. 每次输出时只允许包含一个代码块；如果需要展示多个示例，应将其合并到同一个代码块中，或改为“一个代码块 + 代码外解释”的形式。

## 注释要求

代码中的注释应尽量覆盖以下几类：

- 语法注释：解释某个语言结构、关键字、函数签名或数据结构的写法。
- 逻辑注释：解释某一步为什么这样写，它与前后步骤的关系是什么。
- 业务注释：解释该代码块在整体目标中承担什么职责。
- 数学注释：如涉及公式、损失函数、相似度、梯度等，应补充数学含义。
- 取舍注释：说明为什么选这个写法，而不是另一个更复杂或更底层的实现。

## 生成流程

1. 先识别用户要理解的是“语法”、“机制”还是“业务流程”。
2. 选取最合适的代码表达形式，例如伪代码、Python 示例、类结构示意、函数流程示意等。
3. 先写出最小可理解示例，再逐步补充必要细节。
4. 为关键变量、函数、分支和返回值添加解释性注释。
5. 如果某段逻辑与理论概念相关，补充概念到代码的映射说明。

## 推荐输出结构

生成内容时，建议采用以下结构：

1. 先用一小段中文说明代码在表达什么。
2. 给出完整代码示例。
3. 在代码后补充分步骤解释。
4. 如果有必要，补充“真实工程里还会增加哪些内容”。

## 代码风格建议

- 默认优先使用容易阅读的 Python 示例。
- 命名尽量语义化，避免无意义的单字母变量。
- 如果某个变量必须用数学符号对应命名，应在注释中说明含义。
- 对关键中间结果要写清楚输入、输出和形状（shape）。
- 如果某段代码是简化版，应明确说明简化了哪些现实细节。

## 数学相关要求

如果代码对应的是机器学习（Machine Learning, ML）或深度学习（Deep Learning, DL）机制，应尽量补充：

- 输入张量（Tensor）的含义与维度。
- 核心公式与代码的对应关系。
- 损失函数（Loss Function）在代码中体现在哪里。
- 参数更新（Parameter Update）与梯度（Gradient）的关系。
- 哪些实现细节被省略了，真实框架里如何处理。

## 示例

### 示例一：算法流程型代码

如果用户说：

“给我一个带详细注释的注意力机制代码 sample。”

则输出应类似于：

```python
import math

def scaled_dot_product_attention(query, key, value):
    # query、key、value 可以理解为三个矩阵：
    # query: 当前 token 想“关注什么”
    # key: 每个 token 提供“我有什么特征可供匹配”
    # value: 每个 token 真正携带、最终会被聚合的信息

    # 这里假设 query 的最后一个维度就是 d_k
    # 在注意力公式里，d_k 表示 key 向量的维度大小
    d_k = query.shape[-1]

    # 第一步：计算 query 和 key 的相似度分数
    # key.transpose(-2, -1) 表示交换最后两个维度，以便进行矩阵乘法
    scores = query @ key.transpose(-2, -1)

    # 第二步：进行缩放，避免分数过大导致 softmax 过于尖锐
    scores = scores / math.sqrt(d_k)

    # 第三步：把分数变成概率分布
    # softmax 后，每一行都表示“当前 query 对所有 key 的关注权重”
    weights = softmax(scores, dim=-1)

    # 第四步：用注意力权重对 value 做加权求和
    # 输出表示每个 query 最终聚合得到的上下文表示
    output = weights @ value

    return output
```

然后应继续补充：

- 这段代码对应了哪一个数学公式。
- 为什么要缩放。
- `weights @ value` 的业务含义是什么。
- 真实框架里还可能加入 mask、batch 维、multi-head 等结构。

### 示例二：业务逻辑型代码

如果用户说：

“给我一个智能体读取任务、规划步骤、调用工具的 sample。”

则代码应着重解释：

- 输入任务如何被解析。
- 规划器如何生成步骤。
- 工具调用如何与主流程耦合。
- 为什么要记录中间状态与错误处理。

## 禁止事项

- 只给裸代码，不给解释。
- 注释过于稀少，无法帮助学习。
- 用过度复杂的工程结构掩盖核心逻辑。
- 明明是为学习服务，却把重点放在运行细节而不是理解机制上。
